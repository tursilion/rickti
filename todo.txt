Oh! Except we are caching 3 bitplanes, so it's 219 bytes. To save 26 frames would be 5694 bytes. There is enough in 2k sprite mode if I split the data between the gap at 0x0800 and the GPU RAM at 0x4000 (5694 is 0x163E bytes. Split it 5k below and 0.5k above and we're golden.) Cutting the sprites like this will make the GPU code a little more complex, but it should be more than fast enough to make up for it.

For f18a build I am leaning towards a separate program, since we have to double to 256k anyway we'll have a lot of room.

I think maybe write a different crt0 that uses just one page as a bootstrap. That page, in this case, tests for F18A and loads one or the other program to RAM and jumps to it. Then we can simplify the program CRT to not require a loader or test 32k, saving a few bytes of ROM (since we'll need that one twice). That way we can test before we need to copy anything to RAM.

Bootloader ROM:
- test for 32k present, report if not
- test for F18A present
    - load first program if not (9918A version)
    - load second program if so (F18A version)

The banking for F18A title page and tile sets are split across multiple banks, unlike the 9918A code. This was done to squeeze everything into the spare space in a 256k cart instead of going to 512k, when we clearly had space.

At this point the F18A build and the 9918A build have compatible layouts, EXCEPT the loader banks - both use the first three banks. The expectation was that I would be able to use my vgmcomp2 packer to pack the two programs together into a compressed stream, and have them fit in that 24k along with an unpacker (which also detects F18A). It's a tall order, but I think it's feasible as a lot of the code is shared and should overlap.

vgmcomp2 -pack currently expects only a single stream, so I will need to add support for a second one - if only to get a table telling me where it starts!
