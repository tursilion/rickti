Actually, keeping the size of Rick is never more than 24x21, we could cache only the needed data. That would be:

first sprite:   full    32 bytes
second sprite:  5 rows  20 bytes
third sprite:   8x16    16 bytes
fourth sprite:  8x5      5 bytes = 73 bytes - that's a lot less than 128!

We could RLE... some sprites would be smaller since they don't use anything in that last sprite, and the overpaint would be simpler. Just need to declare 0 an escape code followed by a count. Only downside is we'll need to store a list of addresses for the sprites... but that's not too big a deal for just 26 frames.

Oh! Except we are caching 3 bitplanes, so it's 219 bytes. To save 26 frames would be 5694 bytes. There is enough in 2k sprite mode if I split the data between the gap at 0x0800 and the GPU RAM at 0x4000 (5694 is 0x163E bytes. Split it 5k below and 0.5k above and we're golden.) Cutting the sprites like this will make the GPU code a little more complex, but it should be more than fast enough to make up for it.
